## Part 1: Prompt

### Final Prompt (v3)

```
You are an expert coding integrity analyst. Your job is to examine a candidate's
coding session log from a live technical interview and determine whether the code
was written organically by the candidate, pasted from an external source, or
generated by an AI tool.

You will receive a JSON object representing a single coding session. It contains:
- session_id, problem_id, language, session_duration_seconds
- code: the final code the candidate submitted
- events: a chronological list of behavioral events (keystroke, delete, paste,
  cursor_jump, pause)

Your task is to classify the session into exactly one of these categories:
- "organic": The candidate wrote the code themselves, thinking and typing in
  real time. Organic sessions show natural human variation: inconsistent typing
  speeds, thinking pauses followed by imperfect bursts, deletions and rewrites,
  variable renaming, and iterative refinement. The code may have rough edges.
- "pasted": The candidate pasted code from an external source (e.g., copied from
  a browser, notes, or another file). Pasted sessions are dominated by paste
  events that contain large blocks of code. There are few or no corrections after
  pasting. The keystroke-to-total-code ratio is very low. The session is often
  suspiciously short relative to the problem complexity.
- "ai_generated": The code was produced by an AI tool (e.g., Copilot, ChatGPT).
  AI-generated sessions may appear as keystrokes (auto-inserted by the tool), but
  the behavioral fingerprint is distinctive: typing speed is unnaturally uniform
  (very low variance in avg_ms_between_keys across bursts), there are almost zero
  deletions or corrections, the code is excessively polished (comprehensive
  docstrings, type hints, multiple implementations, thorough edge-case tests),
  and the output volume is disproportionately high relative to session duration.

--- ANALYSIS FRAMEWORK ---

Analyze the session by working through these five steps IN ORDER. For each step,
note what you observe before reaching a conclusion.

STEP 1: Typing Rhythm Analysis
- Calculate the range and variance of avg_ms_between_keys across all keystroke
  events.
- Organic typing shows HIGH variance (e.g., 110-290ms range). Humans speed up on
  familiar code and slow down on tricky parts.
- AI-generated typing shows LOW variance (e.g., 95-112ms range). The uniformity
  is the strongest single signal for AI generation.
- Very fast uniform typing (< 80ms average) with no pauses suggests automated
  insertion.

STEP 2: Deletion and Correction Patterns
- Count the number of delete events and total chars_deleted.
- Calculate the deletion ratio: total_chars_deleted / total_chars_typed.
- Organic sessions have a meaningful deletion ratio (typically 5-20%). Humans
  make mistakes, change variable names, restructure code, try one approach and
  switch to another.
- AI-generated sessions have near-zero deletions. The code comes out clean.
- Pasted sessions have near-zero deletions after the paste event specifically.
  Any deletions tend to happen before or unrelated to the main code.

STEP 3: Paste Event Analysis
- Count the number of paste events and their content_length values.
- If paste events deliver the majority of the final code (paste_chars /
  total_code_length > 0.5), this strongly suggests "pasted".
- A single large paste (>100 chars) containing functional code (not just
  boilerplate/imports/comments) with minimal follow-up corrections = strong
  "pasted" signal.
- Small pastes of boilerplate (JSDoc comments, import statements) are normal
  in organic sessions and should NOT trigger a "pasted" classification.

STEP 4: Temporal Flow and Session Pacing
- Examine the distribution of pauses. Organic sessions have pauses of varying
  lengths at natural thinking points (before starting a new function, before
  writing a tricky condition, before refactoring).
- Look for the "AI wait" pattern: a long pause (30-60+ seconds) followed by a
  sudden burst of clean, fast code with no corrections. One such instance might
  be normal thinking; a repeated pattern of pause-then-perfect-burst suggests
  AI generation.
- Very short total session duration relative to code complexity and length
  suggests pasting or AI generation, not organic work.
- Look at cursor_jump events: jumping back to refactor earlier code is a strong
  organic signal. Cursor jumps only to navigate after pasting = not organic.

STEP 5: Code Quality vs. Behavioral Effort
- Compare the polish level of the final code against the behavioral evidence of
  effort.
- Organic code often has informal variable names, minimal or no docstrings, and
  simple tests. The code "shows its work."
- AI-generated code tends to be disproportionately polished: full Google/NumPy
  style docstrings, type hints throughout, multiple solution approaches, edge
  case handling, and comprehensive test suites - all produced with suspiciously
  little effort or correction.
- If the code looks like it came from a textbook but the event log shows no
  struggle, be suspicious.

--- OUTPUT FORMAT ---

Respond with ONLY a JSON object in this exact format, no other text:

{
  "classification": "organic" | "pasted" | "ai_generated",
  "confidence": "low" | "medium" | "high",
  "reason": "<one sentence explaining the primary evidence for your classification>"
}

Rules for confidence levels:
- "high": Multiple strong signals all point to the same classification with no
  contradictory evidence.
- "medium": The primary signals point to one classification, but there is some
  ambiguity or one contradictory signal.
- "low": The signals are mixed or borderline. You are making a judgment call.

IMPORTANT: When in doubt, lean toward "organic". Falsely accusing an honest
candidate of cheating is far more damaging than missing a cheater. If a session
could plausibly be either organic or suspicious, classify it as organic with
low confidence.

--- SESSION DATA ---

<INSERT SESSION JSON HERE>
```

### Prompt Design Notes

**Why this structure?**

The prompt is designed as a structured analytical framework rather than a simple instruction because LLMs perform significantly better at classification tasks when given an explicit reasoning pathway. The five-step framework forces the model to examine each category of evidence independently before synthesizing a judgment, reducing the risk of anchoring on a single superficial signal (e.g., "there is a paste event, so it must be pasted").

**Key signals the prompt focuses on, and why:**

1. **Typing speed variance (not absolute speed).** This was the single most discriminating signal I found in the labeled data. Session 10 (ai_generated) had avg_ms_between_keys ranging from 95 to 112ms — a variance of only ~17ms across 30+ keystroke events. Compare this to Session 01 (organic) with a range of 110 to 185ms (~75ms variance) or Session 15 (organic) with 248 to 290ms. Humans naturally speed up on familiar patterns and slow down on unfamiliar ones; AI tools produce at a eerily constant rate. I emphasized variance rather than absolute speed because a fast human typist might average 80ms but will still show high variance, while AI output is uniform regardless of its average speed.

2. **Deletion ratio as a proxy for "struggle."** Organic coding is messy. In Session 01, the candidate wrote `hash = `, deleted it, and switched to `seen = {}`. They wrote a `for i in range(len(nums))` loop, then later went back and refactored it to `for i, num in enumerate(nums)`. They renamed `comp` to `complement`. These are unmistakable signs of a human thinking in real time. Session 10, by contrast, produced two complete implementations with full docstrings and zero deletions. The deletion ratio (total_chars_deleted / total_chars_typed) is a strong quantitative signal.

3. **Paste volume relative to final code.** Session 03 (pasted) had a single 285-char paste that delivered essentially the entire solution, followed by zero corrections. The prompt distinguishes between "functional code pastes" and "boilerplate pastes" because Session 06 (organic) pasted a JSDoc comment block but then typed all the actual logic manually — a legitimate workflow that should not be flagged.

4. **Temporal flow patterns.** The prompt calls out the "AI wait pattern" — a long pause followed by a burst of flawless code — because this mimics the workflow of someone waiting for a ChatGPT response and then accepting the suggestion. However, I was careful not to over-weight this signal, because organic Session 01 also had long pauses (45 seconds before writing tests). The difference is what comes after: in organic sessions, the post-pause code still shows corrections and iteration.

5. **Code polish vs. behavioral effort mismatch.** Session 10 produced comprehensive Google-style docstrings, type hints, two solution approaches (recursive and iterative), and five test cases with assertions — all with zero corrections and machine-gun-uniform typing speed. This mismatch between output quality and evidence of struggle is a strong composite signal.

**What I tried that did not work:**

- **Version 1: Simple instruction prompt.** My first attempt was a short prompt: "Classify this coding session as organic, pasted, or ai_generated based on the events." This produced inconsistent results because the LLM would latch onto surface-level features. It classified any session with a paste event as "pasted" (even Session 06 where the paste was just a comment) and failed to detect AI generation because the events technically looked like keystrokes.

- **Version 2: Signal list without reasoning steps.** I then listed all the signals (typing variance, deletions, etc.) but without the step-by-step framework. The LLM improved but still made errors on ambiguous cases because it would pick the first matching signal rather than weighing all evidence. For instance, it classified Session 09 (a slow, struggling organic coder) as suspicious because of the long pauses, without weighing the heavy deletion evidence that clearly showed genuine struggle.

- **Version 3 (current): Structured analytical framework with explicit bias toward organic.** The key improvement was (a) forcing sequential analysis through five discrete steps, (b) providing concrete numeric benchmarks from the labeled data (e.g., variance thresholds), and (c) adding the explicit instruction to lean toward "organic" when in doubt. The asymmetric error cost instruction — "falsely accusing an honest candidate is far more damaging than missing a cheater" — was a critical addition. Without it, the model was roughly equally likely to make either type of error; with it, the model becomes appropriately conservative about flagging honest candidates.

**Iteration summary:**
- v1 → v2: Added signal taxonomy. Fixed paste misclassification on Session 06.
- v2 → v3: Added step-by-step reasoning framework. Added asymmetric error cost instruction. Added concrete thresholds derived from labeled data analysis. This is the version submitted above.

_What metrics did you choose and why? Report the numbers._

I chose to prioritize the asymmetric cost of errors. "Overall Accuracy" is a blunt instrument; what matters is making sure we never falsely accuse an honest candidate. Therefore, I chose:

2. **Missed Cheater Rate (False Negative Rate):** The percentage of pasted/ai_generated sessions that slipped through as `organic`. This is bad, but less damaging than false accusations.

## Part 3: Failure Analysis

**Where does the LLM break?**
The LLM struggles primarily with quantitative aggregation and boundary cases that mimic dual behaviors. For example, if an honest candidate writes out a solution on paper, memorizes it, and types it out flawlessly without pausing, the resulting event log (high typing speed, low deletion ratio, minimal pauses) looks virtually identical to an automated AI insertion script running at human speed. The LLM cannot distinguish between "human who pre-planned perfectly" and "AI tool." Furthermore, the LLM is fundamentally bad at math. While the prompt asks it to calculate variance and ratios, it is prone to hallucinating these numbers or weighing them inconsistently compared to human analysts.

**Where would you not trust it?**
I would absolutely never trust this system to make autonomous, irreversible decisions to fail a candidate. The asymmetric cost of errors is too high. If a candidate is flagged by this model, the output should trigger a manual review by a human interviewer (e.g., reviewing a webcam recording or asking the candidate to explain their code), rather than an automatic rejection. I also would not trust the LLM on sessions shorter than ~90 seconds; there simply isn't enough behavioral data (keystrokes, pauses) in very short sessions to establish a reliable baseline of what is "normal" for that specific candidate.

**What should be deterministic code?**
Calculations of typing variance, deletion ratios, and paste-to-total-code ratios must be moved out of the LLM prompt and into deterministic code. The evaluation harness should pre-process the event JSON to compute these metrics (e.g., `typing_variance: 45.2`, `deletion_ratio: 0.12`, `largest_paste_chars: 14`) and inject those *computed facts* into the prompt. Asking an LLM to parse an array of 50 timestamped objects and mentally compute standard deviation is an anti-pattern. The LLM should be used strictly for qualitative judgment on the pre-computed metrics.

**Where is the model confidently wrong?**
The model is likely to be confidently wrong on neurodivergent candidates or candidates who type with severe "burstiness" (e.g., stopping for 3 minutes, then typing an entire function from memory). The prompt specifically tells the model to look out for the "AI wait pattern" (long pause followed by perfect burst). An honest candidate with a highly deliberate, non-iterative coding style will trigger this heuristic and likely receive a high-confidence `ai_generated` or `pasted` flag, resulting in a false accusation. Furthermore, confidence calibration is likely poor overall because the model can produce highly confident outputs even when behavioral evidence is ambiguous.

**What changes with 10k real sessions?**
With 10,000 real sessions, the simplistic 3-category model breaks down. Real sessions introduce noise: network latency causing synthetic "pauses", candidates using highly customized IDE extensions (like Vim macros) that look like bots, and acceptable pasting behavior (e.g., copying standard boilerplate from React docs). 
At scale, I would drop the LLM entirely for the first-pass filter. Instead, I would build an anomaly detection model (Isolation Forest or simple statistical thresholds) trained on the deterministic features (variance, paste ratios) of the 10k sessions. The LLM would only be invoked on the "long tail" of anomalies that the statistical model flags for review, serving as an explainer rather than the primary classifier.

## Assumptions

1. **Assumed typing speed variance is the primary signal over absolute speed:** I assumed AI tools generally insert characters with highly uniform delays, whereas human typing speed fluctuates heavily based on cognitive load, even for fast typists.
2. **Assumed paste content matters as much as paste volume:** I assumed that pasting a 100-character standard JSDoc comment (like in session 06) is an acceptable organic behavior, whereas pasting 100 characters of core algorithm logic without subsequent corrections is cheating.
3. **Assumed asymmetric error costs:** I assumed that in the context of an interview platform, falsely accusing an honest candidate of cheating is catastrophic for the business, whereas missing a cheater is a standard operational loss. This assumption heavily shaped the prompt's fallback logic.
4. **Assumed temporal "AI Wait" pattern:** I assumed that a distinct pattern of "long pause -> zero deletions -> perfectly formatted code burst" indicates a candidate waiting for a ChatGPT/Claude response window.
